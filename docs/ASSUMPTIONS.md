# 需求理解与假设说明

> **项目：** 运单全流程管理系统 V3  
> **依据：** `exam-v3-exception-waybill-approval-改进版.md`  
> **本文档覆盖题目中未给出的精确数值规则和边界情况处理方案**

---

## 一、概述

本文档记录 V3 系统开发过程中遇到的**信息不完整的业务规则**，以及我作为工程师所做的**假设、判断和设计依据**。题目中故意不给出的精确数值（如审批金额阈值、超时时长等）均在此说明。

**核心原则：** 业务规则的"留白"不是题目疏漏，而是有意为之——现实中产品经理的需求经常不完整，工程师需要主动判断、主动假设。

## 二、9 项留白规则说明

### ① 分级审批金额阈值

| 审批层级 | 金额范围 | 说明 |
|----------|---------|------|
| 一级审批 | > ¥0 且 ≤ **¥5,000** | 所有异常工单先进一级审批 |
| 二级审批 | > **¥5,000** | 金额超过阈值自动升级到二级审批 |

**设定依据：**
- 参考中小型电商物流行业惯例，单票货值 **¥5,000** 是一个常见的分水岭
- ¥5,000 以下属常规异常，一级审批人（仓库主管）可自主决定
- ¥5,000 以上涉及金额较大，需要更高级别审批人介入
- 阈值在 `ApprovalLevelConfig` 表中可配置（`level=1` 的 `maxAmount` 字段），后台可调

**可配置实现：** 不硬编码，通过 `getLevel1MaxAmount()` 函数从数据库表读取。

---

### ② 审批超时时长

| 审批层级 | 超时时长 | 超时后动作 | 代码实现 |
|----------|---------|-----------|---------|
| 一级审批 | **24 小时** | 自动升级到二级审批 | `currentLevel=1` → `level2_approving` |
| 二级审批 | **48 小时** | 自动驳回（timeout_auto_rejected） | `currentLevel=2` → `timeout_auto_rejected` |
| 待审批超时 | **24 小时** | 自动驳回 | `pending_approval` → `timeout_auto_rejected` |

**设定依据：**
- 一级审批 24h：覆盖一个工作日 + 缓冲期，仓库主管通常当天能处理
- 二级审批 48h：金额较大，给更长时间决策，但不能无限拖延
- 超时动作差异：一级超时升二级（给更高级别机会），二级超时直接驳回（无人能再升级）
- 时长在 `ApprovalLevelConfig` 表中可配置

**自动触发机制：** 采用**惰性检测**策略——每次用户访问 `/api/approval`（待审批列表）时，自动检查所有超时工单并更新状态。

---

### ③ 重新提交（重提）次数上限

| 项目 | 值 |
|------|-----|
| 最大拒绝次数 | **3 次** |
| 超过上限后动作 | 自动关闭（状态变为 `timeout_auto_rejected`） |
| 实现字段 | `Ticket.rejectCount` + `Ticket.maxRejectCount` |

**设定依据：**
- 首次提交 + 3 次重提 = 总共最多 4 次机会
- 超过 3 次仍被拒绝，说明问题无法通过当前审批路径解决，需人工介入
- 3 次是一个合理的平衡点：过少不够灵活，过多浪费审批资源
- 拒绝次数在 API 层自动递增（`ticket.rejectCount + 1`），当 `>= maxRejectCount` 时自动关单

---

### ④ 物流异常类型 → 下游动作映射

| 异常类型 | 下游动作 | 是否赔付 | 赔付方向 | 库存联动 | 代码映射 |
|----------|---------|---------|---------|---------|---------|
| 丢件 (lost) | 赔付 + 补发 | ✅ | 赔付客户 (`to_customer`) | 扣库存 | `LOGISTICS_ANOMALY_ACTIONS.lost` |
| 破损 (damaged) | 赔付 + 补发 | ✅ | 赔付客户 (`to_customer`) | 扣库存 | `LOGISTICS_ANOMALY_ACTIONS.damaged` |
| 客户拒收 (rejected) | 退货入库 + 补发 | ❌ | — | 退货入库+扣库 | `LOGISTICS_ANOMALY_ACTIONS.rejected` |
| 超时未签收 (timeout) | 联系客户确认 | ❌ | — | 不变 | `LOGISTICS_ANOMALY_ACTIONS.timeout` |
| 地址错误 (wrong_address) | 重新发货 | ❌ | — | 扣库存 | `LOGISTICS_ANOMALY_ACTIONS.wrong_address` |

**设计原则：**
- 丢件/破损涉及货损，需要赔付客户 + 重新发货
- 拒收只需退货入库后重新发货，不涉及赔付
- 超时只做联系客户，无库存/赔付联动
- 地址错误只需重新发货（改地址），不赔付
- 具体映射在 `src/lib/types.ts` 的 `LOGISTICS_ANOMALY_ACTIONS` 中定义

**品控类异常映射：**
| 异常类型 | 下游动作 | 赔付方向 |
|----------|---------|---------|
| 数量不符 | 供应商追偿 + 调整库存 | `from_supplier` |
| 外观破损 | 供应商追偿 + 退货 | `from_supplier` |
| 规格不符 | 退回供应商 | — |
| 标签错误 | 重新贴标 | — |
| 批次异常 | 退回供应商 + 追偿 | `from_supplier` |

### ⑤ 角色权限划分

| 角色 | 可操作范围 | 权限级别 |
|------|-----------|---------|
| **上报人 (reporter)** | 上报异常、查看自己上报的工单 | 最低 |
| **一级审批人 (level1_approver)** | 审批金额 ≤¥5,000 的工单 | 中 |
| **二级审批人 (level2_approver)** | 审批所有金额的工单 | 高 |
| **品控主管 (qc_supervisor)** | 快速放行、查看所有品控工单 | 中（制衡） |
| **管理员 (admin)** | 全部操作（含转交、配置管理） | 最高 |

**权限规则（均在后端 API 层校验）：**
- **自批自核保护**：`canSelfApprove(ticket.reporterId, approverId)` — 上报人不能审批自己提交的工单
- **层级匹配**：`canApproveLevel(userRole, level)` — 一级审批人只能审批第1级，二级才能审批第2级
- **品控主管限制**：`canFastRelease(userRole)` — 仅 `qc_supervisor` 和 `admin` 可快速放行
- **管理员兜底**：`isAdmin(userRole)` — 管理员可执行转交操作

**多仓库扩展（单租户假设）：**
- `User.warehouse` 字段已预留，当前所有用户 warehouse 为 null（单租户）
- 如需支持多租户：上报/审批时增加 `where: { warehouse: currentUser.warehouse }` 过滤
- 表结构调整：`Ticket` 表加 `warehouse` 字段，查询时自动追加 warehouse 过滤条件

### ⑥ V2 数据同步频率与一致性策略

| 维度 | 策略 | 实现方式 |
|------|------|---------|
| 同步时机 | **按需实时 + 定时兜底** | 每次上报调用 V2 API 校验 + 手动触发批量同步 |
| 实时触发 | 发起异常上报时，实时调用 V2 接口校验运单存在性 | `POST /api/tickets` → `verifyWaybill()` |
| 定时兜底 | 每 **30 分钟**定时批量同步 | Vercel Cron Jobs / 用户手动触发 `/api/sync` |
| 新鲜度标注 | 前端标注"实时获取自 V2"或"使用缓存，同步于 XX:XX" | 工单详情页运单信息区顶部标注 |
| 一致性处理 | 实时校验覆盖策略 | 关键操作（上报/扫描）必调实时接口，非关键展示可用缓存 |

**边界情况处理：**
| 场景 | 处理方式 |
|------|---------|
| V2 服务不可用 | 基于本地快照继续展示，标注"数据可能非最新，获取自 XX 时间" |
| V2 运单金额变更 | 上报时实时校验取最新值，更新本地快照 |
| V2 接口超时/失败 | 5s 超时 + 2 次重试（指数退避 1s→2s） |
| V2 完全崩溃 | 双 URL 降级 + 5 分钟本地缓存兜底 |

**Request ID 链路追踪：**
- 每次 V3 调用 V2 接口生成唯一 `Request ID`（格式：`v3-<base36ts>-<6位随机>`）
- 写入 `SyncLog` 表，包含：调用时间、接口名、请求参数摘要、响应状态码、耗时、错误信息

### ⑦ 品控暂扣超时时长（独立于审批超时）

| 项目 | 值 |
|------|-----|
| 品控暂扣超时时长 | **4 小时** |
| 审批超时时长（一级） | 24 小时 |
| 独立理由 | 压仓成本驱动，远短于审批超时 |

**独立于审批超时的理由：**
- 品控暂扣意味着货物被锁定在仓库中，**无法出库**
- 每多一小时都会产生**仓储成本**（压仓），影响仓库周转率
- 审批超时关注的是"决策效率"（24-48h），品控暂扣关注的是"货物占用成本"
- 两者性质不同，必须使用不同的配置值，不能共用

**超时后处理：**
- 品控暂扣超时（4h 未处理）后，**强制自动进入二级审批**（而非自动放行）
- 理由：超时意味着一级审批人未及时处理，需要更高级别介入
- 此逻辑通过 `QcRule.autoLevel` 字段实现（`autoLevel=2` 时直接进二级审批）

---

### ⑧ 品控规则触发阈值

| 异常子类型 | 触发条件 | 阈值 | 对应审批层级 | 说明 |
|-----------|---------|------|------------|------|
| 数量不符 | 差异率 > | **10%** | 一级 | 差异超过 10% 判定为异常 |
| 外观破损 | 描述包含 | "破损" | 一级 | 关键词匹配 |
| 规格不符 | 描述包含 | "不符" | **二级** | 规格问题较严重，直接进二级 |
| 标签错误 | 描述包含 | "错误" | 一级 | 低严重度 |
| 批次异常 | 描述包含 | "临期" | **二级** | 临期品直接进二级审批 |

**设定依据：**
- 数量差异 **10%** 是行业常见质检标准（±10% 属正常波动范围）
- 规格不符和批次异常直接进二级审批，因为涉及供应商追偿等复杂操作
- 所有阈值在 `QcRule` 表中可配置（`condition` 字段为 JSON：`{"field":"qty_diff","operator":">","threshold":0.1}`）
- 规则执行过程可追溯（记录 `hitRuleId` 和 `hitRuleDesc`）
- 规则按严重度权重排序执行（`critical > high > medium > low`）

### ⑨ 品控主管角色权限边界

| 权限项 | 允许 | 说明 |
|--------|------|------|
| 误判快速放行 | ✅ | 认定扫描结果为误判时，直接解锁批次+关闭工单，绕过审批 |
| 查看所有品控工单 | ✅ | 包括各层级审批中的工单 |
| 查看物流类工单 | ❌ | 只能操作品控相关工单（`source=scan_auto`） |
| 审批金额 >¥500 的工单 | ❌ | 与审批人权限分离，形成制衡 |
| 修改品控规则 | ✅ | 可启用/禁用规则（删除需管理员权限） |
| 转交工单 | ✅ | 审批人离职/禁用时可转交 |

**设计原则：**
- 品控主管的定位是"品控质量把关人"，**不是"审批人"**
- 能快速放行（解决误判问题）但不能独立审批大额工单（需审批人决定）
- 能查看但不能修改审批结果（制衡）
- 操作必须留痕（`ApprovalRecord` 中记录 `action=fast_release` + 复核原因）
- 一个人可以兼任多个角色（如同时是 level1_approver 和 qc_supervisor），操作时按角色判定

## 三、其他设计决策

### 3.1 两套状态机分离设计

| 维度 | 工单状态机（Ticket） | 扫描批次状态机（ScanRecord） |
|------|-------------------|-------------------------|
| 状态字段 | `Ticket.status` | `ScanRecord.batchStatus` |
| 状态值 | 9 种（见 3.3 状态机图） | 3 种（normal / locked / unlocked） |
| 关联方式 | — | `ScanRecord.ticket_id → Ticket.id` |
| 设计原因 | 品控暂扣与审批流程解耦，扫描锁定不应阻塞审批 |

### 3.2 赔付方向字段设计

- `PaymentRecord.direction` = `to_customer`（赔付给客户）/ `from_supplier`（向供应商追偿）
- 物流异常 → 赔付客户（货损理赔）
- 品控异常 → 向供应商追偿（来货质量问题）
- 使用同一张表但方向字段区分，对账逻辑不混用

### 3.3 幂等性设计

| 维度 | 措施 |
|------|------|
| 审批操作 | `token` 唯一令牌 + `@@unique([ticketId, token])` 数据库约束 |
| 扫描录入 | 同一 waybillId+skuCode 存在未关闭工单时，只追加记录不新建工单 |
| 库存/赔付 | 在同一个 `$transaction` 中完成审批状态变更 + 联动操作 |

### 3.4 老系统二开意识

**如果 V2 原本没有对外接口：**
- URL 版本策略：`/api/v2/orders/:id`，后续升级用 `/api/v3/`
- 字段兼容：新增接口字段使用 optional（可空），不破坏已有调用方
- 灰度上线：先在 V2 的 staging 环境部署接口，V3 对接测试后再上生产
- 不影响 V2 现有功能：新增 API 路由是独立的，不修改 V2 现有路由

**V2 接口字段升级（如金额从 int → decimal）：**
- V2 侧：旧接口同时返回 `totalAmount`（int）和新字段 `totalAmountDecimal`（decimal）
- V3 侧：优先读取新字段，不存在则读旧字段做兼容
- 监控：通过 `SyncLog` 表监控接口字段变化

## 四、边界情况与异常处理

### 4.1 并发冲突
- 两个审批人同时打开同一工单，只能一个操作生效
- 第二个提交者收到提示"该工单已被处理，请刷新"
- 实现：事务内校验 `current.status !== ticket.status`

### 4.2 审批人离职/账号禁用
- 管理员可通过 `/api/tickets/[id]/reassign` 转交工单
- 转交时校验目标用户存在且角色匹配
- 转交后重新计算超时时间

### 4.3 V2 接口不可用
- 上报/扫描时 V2 接口超时或失败，返回明确错误提示（非白屏）
- 列表展示降级：基于本地快照的"上一次同步数据"继续展示
- 同步页显示失败日志，可手动重试

### 4.4 数据一致性问题
- 审批状态变更 + 库存/赔付联动在同一个 `prisma.$transaction` 中完成
- `executeLinks` 接受外部事务 `tx` 参数，避免嵌套事务导致外键不可见
- 联动失败会回滚整个事务，保证不出现"审批通过但联动没执行"的中间态

## 附：如果能问产品经理的问题清单

1. **品控暂扣超时 4h 是否合理？** 实际业务中货物压仓成本如何计算？是否有 SLA 要求？

2. **两类异常工单能否并存？** 同一运单存在未关闭的品控异常时，能否再发起物流异常上报？当前实现允许并存（不同类型不影响），但不确定业务上是否允许。

3. **¥5,000 阈值的依据？** 是否有历史赔付数据支撑？不同品类（冷链 vs 常温）是否有不同阈值需求？

4. **多仓库/多商户的实际规模？** 是否需要考虑跨租户数据隔离？还是目前仅单租户？当前按单租户设计，如需多租户需调整 warehouse 过滤。

5. **赔付流程是否对接真实财务系统？** 目前仅做记录（PaymentRecord表），不接真实支付。是否需要对接支付网关/ERP？

6. **"超时未签收"的处理？** 超时多久算异常？是否需要区分"联系不上客户"和"客户明确不要了"？当前统一按 timeout 处理。

7. **品控主管能否批量快速放行？** 还是必须逐条操作？当前为逐条操作，如果批量放行需求频繁需增加批量 API。

8. **接口回写 V2 是必须的还是可选？** `POST /api/v2/orders/:id/anomaly-status` 已实现，但 V2 侧是否需要配合展示异常标记？

9. **200 条测试数据的规模是否足够？** 实际业务中日工单量级是多少？如果量级大（>20万），需加 cursor 分页和索引优化。

## 六、反思题（对应考点9）

### 1. 如果题目直接给出了精确的审批金额阈值和超时时长，架构设计会有什么不同？

如果阈值和时长固定：
- 不需要 `ApprovalLevelConfig` 表，直接在代码中用常量定义即可
- `getLevel1MaxAmount()` 和 `getTimeoutHours()` 函数改为返回常量而非查库
- 架构会更简单，但扩展性降低——改阈值需要重新部署

**当前的可配置设计**虽然复杂一点，但更健壮：后台改阈值不用改代码，适合业务规则可能会变的场景。

### 2. 审批状态变更与库存/赔付联动的一致性，最大的风险点在哪里？不用数据库事务怎么处理？

**最大风险点：** 网络超时或数据库故障导致状态已更新但联动没执行，出现"中间态"。

**不用事务的替代方案：**
- **补偿任务（Saga模式）：** 先更新状态，再异步执行联动。如果联动失败，由补偿任务回滚状态
- **对账任务：** 定期扫描"状态=执行中但联动记录为空"的工单，重新触发联动
- **最终一致性：** 接受短时间的不一致，通过对账修复

当前采用**数据库事务**方案是最直接可靠的。

### 3. 面对一句话级别的模糊需求，你倾向于"先问清楚再动手"还是"先做出一个版本再迭代"？

**视情况而定：**
- 如果有明确的产品经理可以问 → **先问清楚**，避免做错方向返工
- 如果无法问到（如本题考试场景） → **先做 MVP 再迭代**，边做边理解需求
- 本题的"留白"设计本身就鼓励"先做出假设、做出版本、再验证"的思路

### 4. 新增第六种异常类型（如"客户投诉服务态度"），表结构和状态机需要改动哪些地方？

**改动非常小：**
1. `src/lib/types.ts` 的 `AnomalyType` 加一个新的联合类型值
2. `LOGISTICS_ANOMALY_ACTIONS` 或 `QC_ANOMALY_ACTIONS` 加一行映射
3. 前端 `ANOMALY_LABELS` 加一条中文标签

**不需要改：**
- 表结构不变（`anomalyType` 是 String）
- 状态机不变（审批流程通用）
- API 不变（通用接口）

改动成本：**约 5 分钟，3 个文件**。

### 5. 工单量从 200 条增长到 20 万条，最先撑不住的环节是哪个？

**最先撑不住的环节依次是：**

1. **列表查询** — 全表 `ORDER BY createdAt DESC LIMIT 20` 在 20 万条数据下需要加 `createdAt` 索引，否则会全表扫描
2. **超时检测** — 当前的惰性检测每次查 `WHERE status IN (...) AND timeoutAt <= NOW()`，需要在 `(status, timeoutAt)` 上建复合索引
3. **统计计数** — `SELECT COUNT(*)` 在 20 万条下变慢，可以用近似计数或缓存

**优化方案：**
- 加索引：`(status, createdAt)`、`(status, timeoutAt)`、`(waybillId)`
- 分页改用 cursor 方式（`WHERE createdAt < ? LIMIT 20`）替代 `OFFSET`
- 超时检测改为定时任务（Vercel Cron）而非惰性检测

### 6. 给 V2 新增对外接口时，如何保证不破坏 V2 现有的调用方？

**原则：新增不改旧**

1. **URL 版本策略**：新增接口全部放在 `/api/v2/` 路径下，不影响 V2 原有的 `/api/imports`、`/api/rules` 等路由
2. **字段向后兼容**：接口返回值使用 `camelCase` 字段名，新增字段使用 optional（可空），已有字段不删不改
3. **灰度上线**：先在 staging 环境部署接口，V3 对接测试通过后再上生产
4. **接口幂等**：每次调用携带 `X-Request-ID`，V2 侧对重复请求返回相同结果

**V2 运单金额字段类型从 `int` 改成 `decimal` 的兼容处理：**
- V2 侧：同时返回 `totalAmount`（int，旧字段）和 `totalAmountDecimal`（decimal，新字段）
- V3 侧：优先读取 `totalAmountDecimal`，不存在则读 `totalAmount` 做兼容
- 监控：通过 `SyncLog` 表监控接口字段变化，如果新字段一直为空则告警
